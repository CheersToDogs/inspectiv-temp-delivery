Executive Summary

Gap Sweep Additions: Subsequent probing confirmed 240p/480p/720p HLS streams are all accessible without authentication (1080p not available for tested streams). The JS CDN origin assets-ah.flixcdn.com is directly internet-facing without Cloudflare proxy, running EOL nginx with CVE-2025-23419 (SSL session resumption client-cert-auth bypass, CVSS 5.3). The /api/premium/get-packages endpoint returns full billing processor stack, internal tier IDs, and product signatures beyond what a pricing page requires. No mobile app deep-link infrastructure was found on any domain — side-loaded distribution is the likely mobile delivery mechanism for an adult platform. swagger/docs paths on faphouselive.com were false positives — SPA bundle contained the string "swagger" in minified JS.

UPDATED — Critical Streaming Findings Added: Subsequent probing of vr.faphouselive.com identified an unauthenticated API endpoint (/api/vr/v2/models) returning 8.2MB of JSON containing 7,915 live performer records, each including a direct HLS playlist URL served from edge-hls.doppiocdn.com. The stream access is zero-auth end-to-end: manifest URLs resolve to HTTP 200 with Access-Control-Allow-Origin: * on both the manifest server and the media segment server (media-hls.doppiocdn.com). Critically, this bypass is not limited to public streams — performers conducting private shows (234), group shows (123), and virtual private shows (112), all of which require payment to access through the platform UI, have their full HLS delivery chains exposed identically. The MOUFLON token present in manifests is not enforced as an access control. Additionally, the production HTML of vr.faphouselive.com hardcodes an Amplitude analytics key and a Sentry DSN pointing to sentry.stripchat.com, confirming Stripchat as the underlying platform and exposing error telemetry infrastructure.

A full Phase 1 passive reconnaissance and Phase 2 active unauthenticated assessment was conducted against faphouse.com, faphouselive.com, and their associated subdomain surface (65 + 47 subdomains respectively) under an Inspectiv Statement of Work covering internet-accessible assets only. Both domains sit behind Cloudflare (CDN and WAF), terminate TLS 1.3 with AES-256-GCM, and route cache as DYNAMIC — meaning no sensitive responses are currently being served from Cloudflare's edge cache.

The most significant immediate risk is the faphouse.com domain registration expiring in 14 days (April 15, 2026). Beyond that, the live platform (faphouselive.com) exhibits a materially weaker security posture than the main site: it lacks a Content Security Policy entirely, sets session and A/B-test cookies without Secure, HttpOnly, or SameSite flags, and provides no rate limiting on its /login or /forgot-password endpoints. The main site (faphouse.com) carries a PHP session cookie (PHPSESSID) missing only SameSite, and an overly permissive CSP allowing unsafe-inline and unsafe-eval across both HTTP and HTTPS.

Infrastructure reconnaissance identified two delivery CDN origins: assets-ah.flixcdn.com (faphouse.com JS bundles) and assets.chapturist.com (faphouselive.com Angular/React bundles). A backend API server at api.faphouse.com returns JSON-formatted errors, confirming a separate API tier. The live platform exposes an nginx origin at gate.faphouselive.com without Cloudflare proxy (DNS-only record). Certificate transparency logs reveal staging, dev, test, old, magento, and notably mysql08 subdomains, disclosing internal DB naming conventions. No ForgeRock or OIDC discovery endpoints were reachable; the entitlement model appears to be subscription-token based (SUBSCRIPTION references in chapturist.com JS bundles). No CORS misconfigurations or GraphQL introspection were found. No subdomain takeovers were identified.

Twelve confirmed findings are documented below: one operational critical (domain expiry), one high (no CSP on live platform), five mediums, four lows, and three informational items. All path-bruteforce admin panel hits on faphouselive.com were confirmed as SPA routing false positives. Shodan port data for ports 8080/8443 was confirmed as Cloudflare edge infrastructure — not origin exposure.

F-02 — Complete Absence of Content Security Policy on Live Platform
faphouselive.com  |  https://faphouselive.com/  |  Security Headers
GET https://faphouselive.com/ HTTP/1.1 200 OK Server: cloudflare Headers present: HSTS (max-age=31536000), X-Request-Id Headers ABSENT: Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy Comparison — faphouse.com (same CDN, same infrastructure): Content-Security-Policy: default-src 'self' http: https: ws: wss: *.faphouse.com; script-src 'self' 'unsafe-inline' blob: https://assets-ah.flixcdn.com ... ImpactWithout CSP, the live platform has no browser-level defense against cross-site scripting. Any XSS vulnerability found in faphouselive.com (present or future) would have zero mitigation. RemediationImplement a Content Security Policy via Cloudflare Transform Rules or origin server headers. Start with a permissive policy in report-only mode, then tighten. At minimum: Content-Security-Policy: default-src 'self'; frame-ancestors 'self'; object-src 'none'. Add X-Frame-Options: SAMEORIGIN as a fallback. Add X-Content-Type-Options: nosniff and Referrer-Policy: strict-origin-when-cross-origin.

F-04 — CSP Permits unsafe-inline and unsafe-eval
faphouse.com  |  https://faphouse.com/  |  Security Headers
Content-Security-Policy: default-src 'self' http: https: ws: wss: yoti: * *.faphouse.com; img-src * data: blob:; media-src * blob:; script-src 'self' 'unsafe-inline' blob: https://assets-ah.flixcdn.com www.google.com www.gstatic.com accounts.google.com *.google.com *.amplitude.com www.google-analytics.com ... Directives of concern: script-src: 'unsafe-inline' ← allows inline <script> execution script-src: 'unsafe-eval' ← allows eval(), Function(), setTimeout(string) default-src: * (http: https:) ← overly broad fetch scope
Impactunsafe-inline allows any injected inline script to execute in the context of faphouse.com — negating most XSS protection that a CSP would otherwise provide. unsafe-eval allows execution of strings as code, which is commonly exploited in prototype pollution and second-order injection chains. The combination means CSP is present but provides minimal protection against an XSS finding.
RemediationAudit JS bundles served from assets-ah.flixcdn.com and eliminate inline script usage. Replace unsafe-inline with nonce-based CSP (generate a per-request nonce, embed it in the script-src and in each inline script tag). Replace unsafe-eval by refactoring code that uses eval/Function constructor. This is a non-trivial refactor — target it for the next major front-end release cycle.

F-05 — Session/Tracking Cookies Missing All Three Security Flags
faphouselive.com  |  https://faphouselive.com/  |  Cookie Security
Set-Cookie: ABTest_ab_pls_b_v2_key=N_732; path=/; domain=faphouselive.com; expires=Thu, 01 Apr 2027 00:00:00 GMT Missing: Secure flag ← transmittable over HTTP Missing: HttpOnly flag ← readable by JavaScript (document.cookie) Missing: SameSite attr ← CSRF cross-origin submission possible Note: This is an A/B test cookie. Additional cookies set post-login (session tokens) should be audited in P3 — if they inherit this pattern, severity escalates to High.
ImpactWithout the Secure flag, the cookie can be transmitted over unencrypted HTTP connections (e.g., if a user accesses via http://). Without HttpOnly, any JavaScript execution context (including XSS) can read and exfiltrate cookie values. Without SameSite, the cookie is included in cross-site requests, facilitating CSRF. While an A/B test cookie has limited direct session value, the same misconfiguration pattern on session/auth cookies (likely set post-login) would be critical.
RemediationSet all cookies with Secure; HttpOnly; SameSite=Lax at minimum. For session tokens: SameSite=Strict. Apply globally via Cloudflare Transform Rules (Add Response Header) or at the application server layer. Audit all Set-Cookie headers in authenticated flows during P3.

F-06 — PHPSESSID Cookie Missing SameSite Attribute
faphouse.com  |  https://faphouse.com/  |  Cookie Security
Set-Cookie: PHPSESSID=db0534c53beba1ff5e3105fa3a67b6eb; path=/; domain=.faphouse.com; Secure; ✓ HttpOnly ✓ SameSite: ✗ MISSING Without SameSite, browsers default to SameSite=Lax (modern browsers) or SameSite=None (older). Cross-origin POST requests will include this cookie unless SameSite=Strict is set.
ImpactMissing SameSite on a PHP session ID cookie enables cross-site request forgery in scenarios where state-changing operations are performed via POST requests. An attacker on a third-party site could craft a form that auto-submits to faphouse.com endpoints, with the victim's PHPSESSID included. Impact is gated by whether faphouse.com implements additional CSRF token validation on state-changing endpoints.
RemediationAdd session.cookie_samesite = "Lax" in PHP configuration (php.ini or per-application). For subscription and payment flows, consider SameSite=Strict. This is a one-line PHP config change.

F-07/08 — No Rate Limiting on /login and /forgot-password
faphouselive.com  |  /login, /forgot-password  |  Authentication
Test: 12 consecutive requests to /login (0.2s intervals) Results: [200, 200, 200, 200, 200, 200, 200, 200, 200, 200, 200, 200] No 429 Too Many Requests observed. No Retry-After header. ImpactAbsence of rate limiting on page loads combined with likely absence on auth POST endpoints enables automated credential stuffing, brute-force attacks on performer and user accounts, and enumeration of valid usernames via response-time or content differences. RemediationImplement Cloudflare Rate Limiting rules on /login and /forgot-password (and their corresponding API POST endpoints). Flag and notify on high-volume attempts from single IP ranges.
