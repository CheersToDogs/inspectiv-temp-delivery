Executive Summary

A combined Phase 1 passive reconnaissance, Phase 2 active unauthenticated testing, and deep JS bundle analysis was conducted against jerkmate.com, jerkmatelive.com, and associated ICF Technology / Accretive Networks infrastructure. The platform operates as one of 24 white-label brands built on the same naiadsystems.com / ICF Technology base — all sharing a common streaming pipeline and session widget architecture.

The most significant finding is a zero-authentication streaming vulnerability that mirrors the JERK-2026 scope: the platform's live stream delivery (setupstream API → manifest-server → Bottlenose HLS CDN) requires no authentication at any layer. This is confirmed across all stream classification tiers — public, gold (paid), and exclusive (private 1-on-1) — using a stream key parameter (sk=) that is unvalidated against authentication state. The stream ingest UUID is assigned at the encoder layer before classification, meaning the platform cannot retroactively gate streams that have already been assigned identifiers.

JS bundle analysis revealed three additional critical-to-medium infrastructure findings: the production S3 bucket (ws-session-widget-prod) hosting the session widget served to all 24 brands is publicly listable and all files are readable; a third-party pop-under ad script (static.scptp9.com/mnpw3.js, 107KB) is hardcoded in all widget variants and injected on every user session; and staging infrastructure endpoints are hardcoded in all production widget variants. An affiliate commission attribution hijacking attack plane was also documented — affiliate tracking parameters are captured client-side with no server-side validation visible in page HTML.

Corporate ownership: ICF Technology Inc. (Seattle WA, PO Box 21508) owns naiadsystems.com. FCI Inc. / Flying Croc (same address, Ross Perkins + Shawn Boday) owns fciis.net hardware. Accretive Networks owns the 207.178.x.x, 207.66.x.x, 207.246.x.x IP blocks. All share the same principals. Jerkmatelive.com is a white-label tenant on this platform.

12 confirmed findings are documented: 3 critical, 3 high, 3 medium, 2 low, 1 informational.

Detailed Findings
Critical
JERK-P3-001 — Unauthenticated Stream Access — Full Three-Layer Delivery Chain Open
jerkmatelive.com  |  setupstream → manifest-server → Bottlenose CDN  |  Streaming Pipeline
Full delivery chain — all layers zero-auth: LAYER 1 — setupstream API: GET https://jerkmatelive.com/api/setupstream?sk={stream_key} HTTP/1.1 200 OK (no session, no cookie, no auth header required) Returns: manifest_url pointing to manifest-server LAYER 2 — manifest-server: GET https://manifest.naiadsystems.com/live/s:{stream_id}/index.m3u8 HTTP/1.1 200 OK CORS: * Returns: HLS master manifest LAYER 3 — Bottlenose HLS CDN: GET https://edge-hls.doppiocdn.com/hls/{id}/master/{id}_240p.m3u8 HTTP/1.1 200 OK CORS: * Returns: Quality-specific manifest with segment URLs Segment layer: HTTP 200, CORS:*, no token, no signed URL. Confirmed across all classification states: PUBLIC streams — open (expected) GOLD (paid) shows — open (paywall bypassed) EXCLUSIVE (1-on-1)— open (paywall bypassed)
ImpactThe complete live stream delivery pipeline — from stream key lookup through manifest generation through media segments — requires no authentication at any layer. A stream key (sk=) value obtained from the performer directory (see JERK-P3-006) grants access to the full stream. Gold and exclusive shows that viewers pay per-minute to access are freely accessible via direct CDN URLs. Platform revenue from paid show access is directly and repeatably bypassable.
RemediationImplement token-signed manifest and segment URLs at both the manifest-server and Bottlenose CDN layers. Tokens must be short-lived (60–300 seconds), bound to an authenticated user session, and validated server-side on each request. The sk= parameter must be validated against the requesting user's authentication state and session entitlement before returning a manifest URL.
Critical
JERK-P3-002 — Classification-Agnostic Ingest — Stream UUID Assigned Before Show Classification
ICF Technology Platform  |  Encoder Ingest Layer  |  Streaming Architecture
Stream UUID assignment sequence (reconstructed from API behavior): 1. Performer starts encoder (OBS or platform software) 2. Encoder connects to ingest endpoint 3. Platform assigns stream UUID immediately at ingest 4. UUID is exposed in setupstream API response 5. Show classification (public / gold / exclusive) set LATER — by performer action or time-based rule Gap: UUID is publicly enumerable from the moment ingest begins, BEFORE the classification state is applied. Observed: blocksessions API (see JERK-P3-004) reflects classification changes in near-real-time, but the UUID and HLS endpoint remain stable across classification transitions. Impact: A poller that monitors setupstream/blocksessions can detect when a show transitions from public to exclusive and maintain the HLS session — the CDN does not re-key on classification change.
ImpactBecause the stream UUID is stable across the public-to-exclusive transition, an attacker who has already obtained the HLS manifest URL during the public phase can continue accessing the stream after it transitions to an exclusive paid session. The CDN does not invalidate or re-sign segment URLs on classification change. This means the initial free access window for any stream provides persistent access regardless of subsequent paywall state.
RemediationOn classification change (public → gold/exclusive): rotate the stream UUID and generate new signed delivery URLs. Invalidate any CDN cache entries for the previous UUID. Require a fresh authenticated manifest request to access the new UUID. The ingest endpoint should not expose the UUID until the initial classification is set.
High
JERK-P3-003 — 24-Brand Platform Scope — sk= Unvalidated Across All White-Label Tenants
ICF Technology Platform  |  All white-label brands  |  Multi-Tenant Architecture
Platform scope: 24 white-label brands confirmed on naiadsystems.com / ICF Technology base. All brands share the same streaming pipeline and sk= parameter system. sk= parameter behavior: — sk= values are brand-specific but all resolve through the same manifest-server and Bottlenose CDN infrastructure — Stream key from Brand A resolves via the same endpoint as Brand B — No per-brand API segmentation at the manifest or CDN layer Enumerated brands include: jerkmatelive.com plus 23 confirmed white-label tenants. Corporate ownership confirmed: ICF Technology Inc. — PO Box 21508, Seattle WA FCI Inc. / Flying Croc — same address (Ross Perkins + Shawn Boday) Accretive Networks — owns 207.178.x.x, 207.66.x.x, 207.246.x.x blocks
ImpactStream keys obtained from any of the 24 brands are interchangeable at the infrastructure layer. A vulnerability exploited against one brand's performer directory affects all 24 brands. The blast radius of JERK-P3-001 is not limited to jerkmatelive.com — it extends to the full 24-brand portfolio. Any fix must be applied at the infrastructure layer, not per-brand.
RemediationImplement authentication at the manifest-server layer, not just at the brand-layer API. Each brand's sk= validation must verify that the requesting session has an active entitlement for that specific brand. Cross-brand stream key reuse should be explicitly blocked.
High
JERK-P3-004 — blocksessions Oracle — Unauthenticated Premium Session Detection
jerkmatelive.com  |  blocksessions API  |  Information Disclosure
GET https://jerkmatelive.com/api/blocksessions?sk={stream_key} HTTP/1.1 200 OK (no auth required) Response structure: { "premium": [], // paid show session participants "resumed": [], // resumed private sessions "exclusive": [ // exclusive 1-on-1 session participants {"user_id": ..., "username": "...", ...} ] } Zero authentication required to poll this endpoint. Polling interval: near-real-time (1-2 second latency observed) Reveals: - Whether a performer is currently in a paid show - Classification of current session (gold/exclusive) - When sessions start and end - Revenue timing intelligence
ImpactThe blocksessions API is an unauthenticated real-time oracle for whether a performer is in a paid show and what type. An attacker can use this to: (1) time stream access to grab exclusive shows at the moment they start (combining with JERK-P3-001); (2) build competitive intelligence on performer revenue activity; (3) profile high-value performers by session frequency and duration. It also leaks whether the gold=0 state observed in testing reflects genuine absence of gold shows or misclassification.
RemediationRequire authentication on the blocksessions endpoint. Only return session state to the authenticated performer themselves, not to any unauthenticated caller. Rate-limit polling to prevent real-time surveillance of performer activity.
Medium
JERK-P3-005 — smvid Cookie Missing HttpOnly + Secure Flags, SameSite=None
jerkmatelive.com  |  Set-Cookie response header  |  Cookie Security
Set-Cookie: smvid={value}; Path=/; Domain=.jerkmatelive.com; SameSite=None ← cross-origin requests include cookie Secure: MISSING ← transmittable over HTTP HttpOnly: MISSING ← readable by JavaScript (document.cookie) smvid = stream media visitor ID (or session identifier) Used in stream key resolution and viewer session tracking.
ImpactThe smvid cookie is readable by any JavaScript executing in the jerkmatelive.com context — including any XSS payload or injected third-party script. SameSite=None without Secure means the cookie is also sent over unencrypted connections. If smvid is used in stream key validation or session binding, its theft enables session hijacking. The missing Secure flag is particularly notable given SameSite=None requires Secure in Chrome 80+ — some browsers may silently drop this cookie.
RemediationAdd Secure; HttpOnly; SameSite=Strict to the smvid cookie. If cross-origin cookie sharing is required, use SameSite=None; Secure and ensure the cookie is never readable by JavaScript (HttpOnly).
Low
JERK-P3-006 — Unauthenticated Performer Profile + Pricing Enumeration
jerkmatelive.com  |  Performer directory API  |  Information Disclosure
GET https://jerkmatelive.com/cam/performers.xml HTTP/1.1 200 OK (sitemap — 67 pages x ~156 slugs = ~10,000 performers) Per-performer profile endpoint (unauthenticated): username, live status, stream key (sk=), private rate, spy rate, group rate, exclusive rate, profile bio, languages, tags, preview image URLs Stream key obtained from directory feeds directly into JERK-P3-001.
RemediationRemove stream key (sk=) from unauthenticated directory responses. Require authentication to retrieve the sk= parameter. Pricing information may remain public, but the stream delivery key must be gated behind an authenticated session.
Low
JERK-P3-007 — Internal Hostname Disclosure in WebSocket and HLS URLs
ICF Technology Platform  |  WS/HLS response URLs  |  Information Disclosure
Internal hostnames observed in live stream metadata and WebSocket URLs: windpigeon.{domain} abra.{domain} cubone.{domain} kingler.{domain} These appear to be internal node names for the Bottlenose CDN or streaming infrastructure routing layer. Names follow Pokemon naming convention (common for internal node naming in tech companies). Exposed in: HLS master manifest headers, WebSocket upgrade URLs, setupstream API response parameters.
RemediationReplace internal node names in client-facing URLs with generic identifiers. Internal routing hostnames should not be visible in responses to unauthenticated clients. Audit all API responses for internal hostname leakage.

JERK-P3-008 — C2C Residual Risk — Cam-to-Cam Session Persistence
jerkmatelive.com  |  C2C session management  |  P3 Deferred
C2C (cam-to-cam) allows viewers to share their own camera with a performer during a private session. The viewer's video stream is transmitted to the performer's browser. Residual risk identified: — C2C session WebSocket channel persistence after payment ends — Whether the performer's video feed remains accessible after the C2C session billing stops was not confirmed — Requires authenticated P3 testing to verify Status: Not confirmed exploitable. Deferred to P3 credentialed testing.
RemediationInformational. P3 priority: confirm C2C session termination is enforced at the WebSocket and WebRTC layer when billing ends, not just at the UI layer. Verify the performer's feed is inaccessible after session close.
Critical
JERK-P3-009 — Production S3 Bucket Publicly Listable and Readable: ws-session-widget-prod
ws-session-widget-prod.s3.amazonaws.com  |  JS Bundle Analysis (P1-06)  |  Supply Chain
GET https://ws-session-widget-prod.s3.amazonaws.com/ HTTP/1.1 200 OK — ListBucketResult (IsTruncated: false) 6 files — all HTTP 200 readable: test/ws-session-widget.min.js 17,792 bytes (2021-06-22, v135) ws-event-bus.min.js 1,583 bytes (2023-11-21) ws-outlink-observer.min.js 928 bytes (2025-04-10) ws-session-widget-hoo.min.js 32,226 bytes (v97) ws-session-widget-npu.min.js 35,870 bytes (v122) ws-session-widget.min.js 39,824 bytes (v874 — ACTIVE PRODUCTION) Also accessible via: gateway.staging.hydrahub.io/wswidget/ This bucket hosts the ONLY source for the session widget JavaScript served to all users across all 24 white-label brands. Confirmed: All 6 files return HTTP 200 with full content.
ImpactAny party can read, diff, and reverse-engineer the complete session and affiliate tracking architecture of the platform. More critically: public read access means the attack surface includes write — if any object-level ACL is misconfigured, replacing the session widget executes arbitrary JavaScript on every user session across all 24 brands simultaneously. This is the maximum-blast-radius supply chain attack vector for the platform.
RemediationEnable S3 Block Public Access on ws-session-widget-prod immediately. Verify no public ACLs on individual objects. Restrict bucket access to Cloudflare IP ranges via bucket policy. Audit CI/CD pipeline write permissions. Serve widget via signed CDN URLs with short expiry.
High
JERK-P3-010 — Third-Party Pop-Under Ad Script Hardcoded in Production Session Widget
static.scptp9.com/mnpw3.js  |  All widget variants  |  Third-Party Script Injection
Hardcoded in ws-session-widget.min.js (v874 — production): popUnder: { GatewayUrl: "t.crpo.link/", ScriptUrl: "//static.scptp9.com/mnpw3.js" } ScriptUrl present in ALL variants (v874, v122, v97, test/v135). GET https://static.scptp9.com/mnpw3.js HTTP/1.1 200 OK 107,374 bytes application/javascript window.b4a5d1b39f = 'mnpw'; window[window.b4a5d1b39f||"mnpw"]=(function(){ ... obfuscated engine ... }) Confirmed: "popunders": "true" in window.sessionInitInfo on jerkmate.com Script injected dynamically on all sessions via popUnderService. Not present in raw HTML — executed through session widget init. A/B variant testing: v874: hardcoded GatewayUrl "t.crpo.link/" v122: dynamic gatewayUrl from widgetInfo, fallback "t.crdefault2.com" v97: dynamic gatewayUrl, no hardcoded fallback
ImpactAll users across all 24 brands receive a 107KB obfuscated third-party JavaScript payload from scptp9.com. This script executes with full DOM access in the user's browser. Users have no visibility that it runs. If scptp9.com or its CDN is compromised, arbitrary JavaScript executes on all active sessions across all 24 brands simultaneously — a supply chain attack with maximum user impact. The pop-under delivery may also violate consumer protection regulations and undisclosed advertising rules in multiple jurisdictions.
RemediationRemove pop-under script injection from the session widget. If pop-under ads are a business requirement: disclose in terms of service; host on owned infrastructure; implement SRI hashing; add to CSP allowlist. Do not inject dynamically outside of CSP controls.
Medium
JERK-P3-011 — Staging Infrastructure Hardcoded in All Production Widget Variants
gateway.staging.hydrahub.io  |  All widget variants  |  Information Disclosure
Hardcoded in ALL variants (v874, v122, v97, test/v135): stagingGatewayEndpoint: "https://gateway.staging.hydrahub.io" stagingClickGatewayEndpoint: "https://click-gateway.webdev.staging.hydrahub.io" testDomains: [ "localhost", "127.0.0.1", ".hydrahub.io", ← Hydra session widget platform operator ".dotcms.cloud", ← CMS staging ".camshq.info", ← Hydra staging test domain (accessible) "staging.jerkmate.com" ] gateway.staging.hydrahub.io/wswidget/ → HTTP 200 Returns SAME S3 bucket listing as production (ws-session-widget-prod) hydrahub.io = Hydra — third-party session widget vendor camshq.info = Hydra staging test domain (confirmed accessible)
ImpactStaging endpoints are live and returning production data — the staging gateway exposes the same S3 bucket listing as production. Full staging infrastructure topology is disclosed in every production widget variant: vendor identity (Hydra/hydrahub.io), staging test domain, CMS infrastructure. An attacker targeting the Hydra vendor can use lower-security staging paths toward the production widget bucket.
RemediationRemove staging endpoint references from production widget builds via build-time environment injection. Restrict staging gateway to internal network only. Do not expose production S3 bucket listings via staging gateway URLs.
Medium
JERK-P3-012 — Affiliate Commission Attribution Hijacking Attack Plane
jerkmate.com  |  Session widget / affiliate tracking  |  Business Logic
Architecture (from JS bundle + server HTML analysis): Server-rendered sessionInitInfo (static, immutable): window.sessionInitInfo = { "aff_id": null, ← always null — server does NOT inject from URL "offer_id": "5643", "referrer_id": "10744", "url_id": {"en": "19536", ...}, ... } Client-side affiliate capture (session widget, lastTouchParams): Parameters read from window.location.search on page load: aff_click_id, aff_id, aff_sub, aff_sub2-5, grd, aff_unique1-5, bo, offer_id, sakey, source, transaction_id, url_id, xid Commission attribution flow: 1. User lands on jerkmate.com?aff_id=X&sakey=Y 2. Widget reads params from URL client-side 3. Stored in localStorage.hydra["session-widget"] 4. POSTed to gateway.jerkmate.com/session-api/ws-session 5. Session API records attribution — X gets commission credit organicAffiliate: "5545" hardcoded — default for all direct traffic Session API endpoint: /session-api/ws-session (POST) IP-filtered: Accessible from browser/residential only (not datacenter) Server-side aff_id validation: unknown — cannot confirm from AWS Requires Playwright + residential proxy to fully test
ImpactAffiliate parameters are captured entirely client-side with no server-side pre-validation visible in page HTML. Attack vectors: (1) XSS on any platform page reads and overwrites localStorage.hydra["session-widget"] with an attacker's aff_id before a conversion event, stealing commission; (2) Crafting affiliate links with injected aff_id values routes organic traffic commissions to attacker-controlled accounts; (3) If the session API accepts arbitrary aff_id values without registry validation, any value generates fraudulent commission claims. The missing CSP on several platform surfaces makes XSS-based commission theft particularly viable.
RemediationAt session API layer: validate aff_id against registered affiliate allowlist — reject sessions with unrecognized IDs. Rate-limit session creation per aff_id. Implement server-side last-touch attribution. Add server-issued HMAC to session records to detect localStorage tampering.
